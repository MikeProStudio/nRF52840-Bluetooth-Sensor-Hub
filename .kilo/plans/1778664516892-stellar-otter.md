# Plan: PDM Audio Stream via BLE (Branch `pdm-stream`)

## Ziel
nRF52840 Sense als drahtloses Mikrofon für einen PC-KI-Assistenten: 16kHz/16bit/Mono-Audio via BLE GATT Notifications an einen PC streamen.

## Architektur-Entscheidungen

### Warum ADPCM statt Raw PCM?
- Raw PCM: 16kHz × 16bit = **256 kbps = 32 KB/s**
- BLE 2M PHY, Connection Interval 15ms, ~6 Pakete/Event, 251 Bytes/Paket → theoretisch ~100 KB/s
- **Praktisch** mit GATT-Overhead (ATT + L2CAP + ACL Header = ~10 Bytes): ~67 KB/s
- Pro Notification: ~241 Bytes Nutzlast → 120 PCM-Samples = 7.5ms Audio
- Benötigt 133 Notifications/s → bei 15ms Interval nur 66/s möglich → **Raw PCM nicht realtime-fähig**
- **Mit ADPCM (IMA/DVI, 4:1)**: 241 Bytes → ~480 Samples = 30ms Audio → nur 33 Notifications/s → **problemlos machbar**

### GATT-Design
- Neuer Characteristic `BT_UUID_AUDIO_STREAM_CHRC` (neue 128-bit UUID) mit `BT_GATT_CHRC_NOTIFY`
- Jede Notification: 1-Byte Header (Block-Sequenznummer für Loss-Detection) + ADPCM-Daten (bis zu MTU-3 Bytes)
- Der existierende `BT_UUID_AUDIO_CHRC` (RMS/ZCR) bleibt für die Pegel-Visualisierung erhalten
- **MTU-Limit (wichtig)**: Mit `BT_CTLR_DATA_LENGTH_MAX=69` liegt das LL-Packet-Limit bei 69 Bytes. ATT_MTU ≈ 65 Bytes, max. Notification-Payload = **62 Bytes**. Jede Notification muss ≤62 Bytes bleiben, sonst schlägt `bt_gatt_notify` stumm fehl.

### ADPCM Format (IMA/DVI)
- Encoder-State pro Stream: `step_index` (0-88), `predictor` (int16)
- Decoder-State identisch, wird mit erstem Block synchronisiert (Header-Flag)
- Block-Header: 1 Byte = `0x80 | seq_nibble(7bit)`, ggf. `0x00` für Reset/Sync
- Jedes Input-Sample (16bit) → 4bit ADPCM-Nibble
- 2 Samples → 1 Byte Output
- **Notification-Aufteilung**: 120 Samples (60 Bytes ADPCM + 1 Byte Header = 61 Bytes) pro Notify — passt in 62-Byte-Limit

### 2. `src/adpcm.h` + `src/adpcm.c` — IMA ADPCM Encoder/Decoder (NEU)

```c
// Zustand
struct adpcm_state {
    int16_t predictor;
    uint8_t step_index;
};

// Encoder: PCM16 → ADPCM4 (zwei 4-bit Nibbles pro Byte)
void adpcm_encode(struct adpcm_state *state, const int16_t *pcm, uint8_t *adpcm, size_t sample_count);

// Decoder: ADPCM4 → PCM16
void adpcm_decode(struct adpcm_state *state, const uint8_t *adpcm, int16_t *pcm, size_t sample_count);
```

- Enthält die IMA/DVI ADPCM Step-Tabelle und Algorithmus
- `adpcm_encode` nimmt ein Array von 16-bit PCM Samples und produziert 4-bit Nibbles (gepackt in Bytes)
- `adpcm_decode` macht das Inverse

### 6. `tools/ble_audio_receiver.py` — Python PC-Empfänger (NEU)

```python
# Verwendet: bleak, numpy (optional, für WAV-Export)
# pip install bleak pysoundfile

import asyncio
from bleak import BleakScanner, BleakClient

AUDIO_STREAM_CHRC_UUID = "12345678-1234-5678-1234-56789abcdef8"

class AudioReceiver:
    def __init__(self, device_name="Skynet AI Beacon"):
        self.device_name = device_name
        self.adpcm_state = {"predictor": 0, "step_index": 0}
        self.audio_buffer = bytearray()  # Roh-PCM-Daten
        self.callback = None  # User-Callback für Audio-Chunks
        
    async def notification_handler(self, sender, data):
        # 1. ADPCM Header parsen (1 Byte: seq + flags)
        seq = data[0] & 0x7F
        is_reset = (data[0] & 0x80) == 0
        
        if is_reset:
            self.adpcm_state = {"predictor": 0, "step_index": 0}
        
        # 2. ADPCM → PCM decodieren
        pcm = adpcm_decode_block(self.adpcm_state, data[1:])
        
        # 3. An Audio-Buffer anhängen / Callback aufrufen
        if self.callback:
            self.callback(pcm)  # bytes oder numpy array
        
        self.audio_buffer.extend(pcm)
    
    async def connect_and_stream(self):
        device = await BleakScanner.find_device_by_name(self.device_name)
        async with BleakClient(device) as client:
            await client.start_notify(AUDIO_STREAM_CHRC_UUID, self.notification_handler)
            await asyncio.sleep(float('inf'))  # Stream forever
    
    def get_audio(self) -> bytes:
        """Gibt den gesamten akkumulierten PCM-Buffer zurück und leert ihn"""
        buf = bytes(self.audio_buffer)
        self.audio_buffer.clear()
        return buf
    
    def save_wav(self, filename, sample_rate=16000):
        """Speichert den Audio-Buffer als WAV-Datei"""
        import wave
        with wave.open(filename, 'wb') as wf:
            wf.setnchannels(1)
            wf.setsampwidth(2)
            wf.setframerate(sample_rate)
            wf.writeframes(self.audio_buffer)
```

```

    → [PC] bleak Client → notification_handler()
    → adpcm_decode() → PCM16-Bytes
    → Audio-Buffer / WAV-Datei / Live-Stream
```

## Risiken / Offene Punkte

- **DMIC-Read Blocking**: `dmic_read()` blockiert bis ein PDM-Block fertig ist (~32ms bei 512 Samples/16kHz). Das ist akzeptabel, da der Thread dediziert läuft.
- **BLE-Buffer-Überlauf**: Wenn der PC nicht schnell genug ACKs sendet, können BLE-TX-Buffer vollaufen. Der Thread muss `-ENOBUFS` von `bt_gatt_notify` behandeln (yield + retry).
- **Koexistenz mit altem audio_thread**: Beide Threads nutzen `dmic_read()`. Entweder wir deaktivieren den alten Thread temporär oder teilen den DMIC-Stream. **Vorschlag**: Den alten audio_thread behalten, aber er liest parallel vom selben DMIC (DMIC unterstützt das via `dmic_read(mic_dev, 0, ...)` Stream-ID 0 — bei nur einem Stream ok). Oder: Ein Thread macht beides.
- **Connection Interval 7.5ms**: Das ist aggressiv und erhöht den Stromverbrauch. Falls die Akkulaufzeit kritisch ist, auf 15ms mit ADPCM (reicht auch).
- **Fix: Ctrl+C → save_wav()**: Zwei Probleme in `tools/ble_audio_receiver.py`:
  - `main()`: `save_wav()` in `try/finally` (bereits erledigt)
  - `connect_and_stream()`: `except KeyboardInterrupt` fängt in Python 3.11 keinen `CancelledError` (Ctrl+C auf `await`). Fix: `CancelledError` mitfangen:
    ```python
    except (KeyboardInterrupt, asyncio.CancelledError):
        print("\nStopped by user.")
    ```
- **MTU-Limit → Notification-Größe anpassen**: `BT_CTLR_DATA_LENGTH_MAX=69` limitiert LL-Packets auf 69 Bytes → ATT_PAYLOAD max. 62 Bytes pro Notification. Aktuelle 129 Bytes passen nicht. Der Printk `seq=... rms=...` bestätigt, dass der Thread läuft, aber `bt_gatt_notify` wegen zu großer Payload stumm fehlschlägt.
  - **Fix in `src/audio_stream.c`**: 
    - `ADPCM_SAMPLES_PER_NOTIFY 256` → `120` (60 Bytes ADPCM + 1 Header = 61 Bytes ≤ 62)
    - `NOTIFY_PAYLOAD_SIZE` → `61`
    - `bt_gatt_notify`-Fehler (außer `-ENOBUFS`) per printk loggen
    - 512-Samples-Block → 4×120 + 1×32 = 5 Notifications pro Block
  - **Fix in `src/audio_stream.c`**: `do { ... } while (notify_ret == -ENOBUFS);` erweitern um:
    ```c
    } while (notify_ret == -ENOBUFS);
    if (notify_ret < 0) {
        printk("AUDIO_STREAM: notify err %d\n", notify_ret);
    }
    ```
- **Audio Gain**: PDM liefert zu niedrigen Pegel. `#define AUDIO_GAIN 8.0f` in `audio_stream.c`. PCM-Samples vor ADPCM-Encoding mit Gain multiplizieren, auf ±32767 clampsen.
- **Reconnect-Bug**: Nach 2 erfolgreichen Verbindungen bricht der 3. Versuch mit "Device not found". Der `bt_le_adv_stop()` im `disconnected`-Callback wird vom Controller nicht sofort verarbeitet, wenn die Verbindung noch nicht vollständig abgebaut ist. Fix in `main.c` `start_advertising()`: 
  - `bt_le_adv_stop()`, Rückgabewert loggen
  - `k_sleep(K_MSEC(50))` nach stop, damit Controller Zustand wechseln kann
  - `bt_le_adv_start()` mit korrekten Named-Constants-Options (nicht rohe Bit-Shifts)
  - Fehler-Rückgabe loggen
- **Error-Spam vermeiden**: `notify err -128` (`-ENOTCONN`) unterdrücken, nur echte Fehler loggen.
- **Battery-Anzeige (0.01V / 1.34V)**: Der Battery-Code (`read_battery_voltage_impl()`) wurde in keiner meiner Änderungen angetastet (identisch zum Original). Das beobachtete Verhalten (0.01V → alle paar Sekunden +0.33V → Zielwert) wird durch den SMA-Filter (Größe 12) verursacht: nach Flashen oder Akku-einstecken sind die ersten Filter-Einträge 0V (BSS), der echte Wert (z.B. 3700mV) wird erst nach 12 Iterationen erreicht.
  - **Fix in `main.c` `read_battery_voltage_impl()`**: Nach der Kalibrierung einen `delta`-Check einbauen. Wenn `abs(val_mv - current_filtered) > 500`, den SMA-Filter zurücksetzen und mit dem neuen Wert initialisieren:
    ```c
    /* SMA Filter */
    int32_t old_filtered = battery_status.voltage_mv;
    adc_history[filter_idx] = val_mv;
    ...
    val_mv = sum / count;

    /* Bei grossem Sprung (>500mV) Filter reset */
    if (old_filtered > 0 && abs(val_mv - old_filtered) > 500) {
        memset(adc_history, 0, sizeof(adc_history));
        filter_idx = 0;
        filter_full = false;
        adc_history[0] = val_mv;
        filter_idx = 1;
        val_mv = old_val;  /* keep old value for display */
    }
    ```
  - Besser: den Rohwert `val_mv` vor dem Filter direkt verwenden, wenn delta > 500mV:
    ```c
    if (old_filtered > 0 && abs(val_mv - old_filtered) > 500) {
        filter_full = true;
        for (int i = 0; i < FILTER_SIZE; i++) {
            adc_history[i] = val_mv;
        }
        filter_idx = 0;
    }
    ```

## Phase 2: Web UI Integration + STT + AI

### Architektur

```
Browser (index.html)
  ↕ WebSocket (Audio-Stream + STT-Ergebnisse)
  ↕ HTTP POST/GET (Gain, Befehle)
  ↕ REST API (Status, Battery, IMU)
Python Backend (tools/server.py, FastAPI + uvicorn + bleak)
  ↕ WebSocket (STT-Text → AI-Chain)
  ↕ GATT (nRF52)
nRF52840 (BLE Peripheral)
```

**Warum Python-Backend statt reinem Web Bluetooth?**
- Web Bluetooth kann nach Disconnect nicht zuverlässig reconnecten
- 133 Notifications/sec überlastet den Browser-BLE-Stack
- STT und AI benötigen Python-Bibliotheken (Vosk/Whisper, Ollama)
- Gain-Regelung via HTTP ist einfacher als via GATT
