# Plan: BLE HID Audio Device + Live Streaming Pipeline

## Overview

Modify the nRF52840 firmware to:
1. Advertise as a Bluetooth HID Audio device (standard service UUIDs) for smartphone pairing
2. Implement a "Live Audio" streaming mode from the PDM mic
3. Add a "Live Audio" button to the WebUI to toggle the stream

## Architecture

The audio pipeline is: **PDM Mic → DMIC → PCM → ADPCM → BLE GATT Notify → Python Backend → WebSocket → WebUI**

Smartphone connection path: **nRF52840 BLE → Smartphone (paired via standard profiles)**

## Changes Required

### 1. `prj.conf` — BLE Audio & HID Configuration

**Add the following Kconfig options:**

```kconfig
# HID over GATT Service (smartphone recognizes as HID device)
CONFIG_BT_HIDS=y
CONFIG_BT_HIDS_REPORT_MAP_MAX_SIZE=256

# Bluetooth Audio Profile (LE Audio)
CONFIG_BT_AUDIO=y
CONFIG_BT_BAP_UNICAST=y
CONFIG_BT_BAP_UNICAST_SERVER=y
CONFIG_BT_PACS=y
CONFIG_BT_ASCS=y
CONFIG_BT_ISO=y
CONFIG_BT_ISO_MAX_CHAN=5

# Audio stack: segmentation required for ISO audio data
CONFIG_BT_AUDIO_TX_SEG_MAX=10

# LE Audio codec config
CONFIG_BT_AUDIO_CODEC_LC3=y

# Increase buffer pools for live audio throughput
CONFIG_BT_L2CAP_TX_BUF_COUNT=100
CONFIG_BT_BUF_ACL_TX_COUNT=80
CONFIG_BT_BUF_EVT_RX_COUNT=80
CONFIG_BT_CONN_TX_MAX=100
CONFIG_HEAP_MEM_POOL_SIZE=49152
CONFIG_MAIN_STACK_SIZE=8192
CONFIG_SYSTEM_WORKQUEUE_STACK_SIZE=8192
```

**Rationale:** Standard Bluetooth Audio service UUIDs advertised via `CONFIG_BT_AUDIO`/`CONFIG_BT_PACS`/`CONFIG_BT_ASCS` cause smartphones to recognize the device as an audio peripheral. HID over GATT enables HID device recognition. Larger buffer pools prevent `-ENOBUFS` during high-throughput live streaming.

#### 2d. Live Audio Control Write Handler

```c
static ssize_t write_live_audio_ctrl(struct bt_conn *conn,
    const struct bt_gatt_attr *attr,
    const void *buf, uint16_t len, uint16_t offset, uint8_t flags)
{
    if (len != 1) return BT_GATT_ERR(BT_ATT_ERR_INVALID_ATTRIBUTE_LEN);
    uint8_t val = ((uint8_t *)buf)[0];
    live_audio_active = (val != 0);
    printk("Live Audio: %s\n", live_audio_active ? "START" : "STOP");
    return len;
}
```

### 4. `src/audio_stream.c` — Live Streaming Mode

The audio encoding format (ADPCM) stays unchanged in both modes for full backward compatibility with the existing Python backend decoder (`ble_client.py`). The `live_audio_active` flag controls whether streaming notifications are sent.

#### 4a. Modify `audio_stream_thread_impl`

In the main processing loop, the existing notification code (around line 226) already sends ADPCM-encoded audio chunks via `bt_gatt_notify`. The `live_audio_active` flag adds explicit on/off control:

- When `live_audio_active == false`: skip the `bt_gatt_notify` calls for audio data (RMS/ZCR level is still sent for PSD visualization)
- When `live_audio_active == true`: send ADPCM notifications as before

The existing notify loop already handles this correctly — we just wrap it in a conditional check:

```c
// In the notify section, check live_audio_active
if (*cfg->live_audio_active) {
    // ... existing ADPCM notification code ...
}
```

**Key design decisions:**
- ADPCM encoding runs in both on/off states (minimal latency to start)
- Only the `bt_gatt_notify` calls are gated by `live_audio_active`
- RMS/ZCR level data (`audio_level_buf`) still sent regardless for PSD chart
- The Python backend already decodes ADPCM → PCM and forwards to WebUI via WebSocket — no changes needed there

#### 6b. Add JavaScript for Live Audio Control

Add to the existing JS (after the gainslider handler, around line 1694):

```javascript
// ── Live Audio Mode (BLE Direct) ──
let liveAudioActive = false;

document.getElementById('live-audio-btn').addEventListener('click', async () => {
    if (!settingsService && !backendWs) return;
    const btn = document.getElementById('live-audio-btn');
    
    if (!liveAudioActive) {
        // Start live audio via BLE (direct GATT write)
        if (settingsService) {
            try {
                const c = await settingsService.getCharacteristic(LIVE_AUDIO_CTRL_CHAR);
                await c.writeValue(new Uint8Array([1]));
                liveAudioActive = true;
                btn.innerText = '⏹ Stop Live Audio';
                btn.style.borderColor = 'var(--accent)';
                btn.style.color = 'var(--accent)';
                document.getElementById('recording-duration').innerText = '🔴 LIVE AUDIO STREAMING';
                console.log('Live Audio: STARTED');
            } catch(e) {
                console.error('Live audio start failed:', e);
            }
        } else if (backendBaseUrl) {
            // Fallback: via backend API
            try {
                const resp = await fetch(`${backendBaseUrl}/api/liveaudio/start`, {method:'POST'});
                if (!resp.ok) throw new Error('Backend live audio start failed');
                liveAudioActive = true;
                btn.innerText = '⏹ Stop Live Audio';
                btn.style.borderColor = 'var(--accent)';
                btn.style.color = 'var(--accent)';
                document.getElementById('recording-duration').innerText = '🔴 LIVE AUDIO STREAMING';
            } catch(e) {
                console.error('Live audio start via backend failed:', e);
            }
        }
    } else {
        // Stop live audio
        if (settingsService) {
            try {
                const c = await settingsService.getCharacteristic(LIVE_AUDIO_CTRL_CHAR);
                await c.writeValue(new Uint8Array([0]));
            } catch(e) {
                console.error('Live audio stop failed:', e);
            }
        } else if (backendBaseUrl) {
            try {
                await fetch(`${backendBaseUrl}/api/liveaudio/stop`, {method:'POST'});
            } catch(e) {
                console.error('Live audio stop via backend failed:', e);
            }
        }
        liveAudioActive = false;
        btn.innerText = '🔴 Live Audio';
        btn.style.borderColor = '';
        btn.style.color = '';
        document.getElementById('recording-duration').innerText = '';
    }
});
```

### 7. `tools/server.py` — Backend Live Audio API Endpoints

Add for fallback control path (when WebUI connects via backend, not direct BLE):

```python
# Track live audio state
_live_audio_active = False

@app.post("/api/liveaudio/start")
async def live_audio_start():
    global _live_audio_active
    if not ble_client.is_connected or ble_client.client is None:
        raise HTTPException(400, "BLE not connected")
    # Write to the live audio control characteristic
    try:
        ch = ble_client.client.services.get_characteristic(LIVE_AUDIO_CTRL_CHAR_UUID)
        if ch:
            await ble_client.client.write_gatt_char(ch, bytearray([1]))
            _live_audio_active = True
            broadcast_event("live_audio", {"active": True})
            logger.info("Live audio started")
            return {"status": "started"}
        else:
            raise HTTPException(404, "Live audio control characteristic not found")
    except Exception as e:
        raise HTTPException(500, str(e))

@app.post("/api/liveaudio/stop")
async def live_audio_stop():
    global _live_audio_active
    if not ble_client.is_connected or ble_client.client is None:
        raise HTTPException(400, "BLE not connected")
    try:
        ch = ble_client.client.services.get_characteristic(LIVE_AUDIO_CTRL_CHAR_UUID)
        if ch:
            await ble_client.client.write_gatt_char(ch, bytearray([0]))
            _live_audio_active = False
            broadcast_event("live_audio", {"active": False})
            logger.info("Live audio stopped")
            return {"status": "stopped"}
        else:
            raise HTTPException(404, "Live audio control characteristic not found")
    except Exception as e:
        raise HTTPException(500, str(e))
```

And add the constant:
```python
LIVE_AUDIO_CTRL_CHAR_UUID = "12345678-1234-5678-1234-56789abcdef9"
```

Also enable the live audio button when BLE connection is established:
```python

```
